# Участие человека (Human-in-the-Loop): Требования и ограничения

## Проблема

Конвейер полностью автономен: точки взаимодействия с человеком от начала до PR отсутствуют.
Процесс ломается, когда:

- **Агенту требуются уточнения.** PM не понимает задачу, Архитектор не может выбрать вариант без контекста, Исполнитель столкнулся с неожиданным ограничением. Сейчас агент галлюцинирует или конвейер завершается ошибкой.
- **Качество страдает без обратной связи.** Агенты принимают решения в вакууме. 1 минута ввода данных человеком на этапе планирования экономит 30 минут переделок в цикле исполнитель+QA.

Фундаментальное противоречие: конвейер должен быть автономным (цель проекта), но иногда нуждается в человеке. Нужен механизм, позволяющий агенту **попросить о помощи** и **дождаться ответа**, не нарушая автономию в остальных случаях.

## Ограничения

- GitHub Issue = единственный пользовательский интерфейс. Терминал = только для отладки.
- Агенты пишут комментарии в issue через `gh` CLI - движок ничего не знает о GitHub.
- Работа скрипта не должна прерываться из-за отсутствия ответа от человека - он должен приостанавливаться и возобновляться, когда ответ будет готов.
- Соло-разработчик: ответ может поступить через минуты или часы.
- Сессия Claude CLI имеет таймаут (по умолчанию ~30 мин в движке).
- `gh api` = бесплатно (лимит 5000 запросов в час для токена авторизации).
- Хук `before` уже поддерживается движком (shell-команда перед агентом).
- Тип узла `human` существует, на данный момент = только запрос в терминале.
- `state.json` + `--resume` уже работают.
- Движок должен оставаться независимым от проекта (никакого специфичного для GitHub кода в ядре движка).

## Требования

- Конвейер ДОЛЖЕН иметь возможность ждать ответа человека без:
  - ручного `--resume`
  - опроса (polling) через сессии Claude (дорого)
  - специфичного для GitHub кода в движке/скриптах
- Агент ДОЛЖЕН иметь возможность задать вопрос и ждать ответа в рамках обычного потока конвейера.
- Механизм ожидания ДОЛЖЕН быть дешевым (никаких вызовов Claude API во время ожидания).
- Ответ человека ДОЛЖЕН быть отличим от комментариев агента.
- Таймаут ДОЛЖЕН настраиваться для каждого узла; при таймауте → узел корректно завершается с ошибкой.
- Несколько узлов с участием человека в одном запуске ДОЛЖНЫ работать независимо.
- Сбой движка во время ожидания ДОЛЖЕН быть восстанавливаемым (следующий запуск обнаруживает приостановленное состояние).
- Пользователь ДОЛЖЕН инициировать возобновление одной командой (в идеале той же `deno task run`).

### ПРОРЫВ: Идея #51 — Перехват `AskUserQuestion` из JSON-потока

**Эксперимент:** `CLAUDECODE="" claude -p "Спроси меня о чем то" --output-format json`

**Результат:** Claude CLI в `-p` (non-interactive) режиме:
1. Имеет `AskUserQuestion` в tool-списке (виден в `init` событии)
2. Агент **естественно вызывает** `AskUserQuestion` когда ему нужен ввод от человека
3. В `-p` режиме вызов **denied** (permission_denials) — но **полностью виден в JSON-потоке**
4. Вопрос приходит **структурированным**: `{question, header, options[], multiSelect}`

**Схема работы:**
1. Движок стримит JSON-вывод claude (уже делает — `onOutput` callback в `agent.ts`)
2. Движок детектирует `tool_use` с `name: "AskUserQuestion"` в потоке
3. Движок извлекает `input.questions` — получает структурированный вопрос с вариантами
4. Движок:
   - Сохраняет вопрос в артефакт (`pending-question.json`)
   - Вызывает внешний скрипт (hook) для доставки вопроса (e.g. `gh issue comment`)
   - Помечает узел как `waiting` в `state.json`
   - Записывает `session_id` для будущего `--resume`
   - Завершает или приостанавливает процесс claude
5. При resume: движок вызывает `claude --resume <session_id>` с ответом человека

**Эксперимент 2: Resume с ответом — РАБОТАЕТ**

```bash
# Шаг 1: claude спрашивает, получает denial, завершается
claude -p "Спроси меня какой мой любимый язык" --output-format json
# → permission_denials содержит структурированный вопрос
# → session_id: bbe8c358-...

# Шаг 2: resume с ответом человека
claude --resume bbe8c358-... -p "Go" --output-format json
# → "Go хорош тем, что его минималистичный дизайн..."
```

Claude при resume **видит контекст** предыдущей сессии (свой вопрос + denial) и правильно интерпретирует "Go" как ответ. Контекст сессии сохранён.

**Итоговая схема HITL через AskUserQuestion:**

```
┌─────────┐    tool_use:AskUserQuestion    ┌─────────┐
│  Agent   │──────────────────────────────→│  Engine  │
│(Claude)  │   (denied, visible in JSON)   │          │
└────┬─────┘                               └────┬─────┘
     │ session ends                              │
     │                                           │ extract question from
     │                                           │ permission_denials
     │                                           ▼
     │                                    ┌──────────────┐
     │                                    │  Hook script  │
     │                                    │ (gh issue     │
     │                                    │  comment)     │
     │                                    └──────┬───────┘
     │                                           │
     │              WAIT (no API calls)          │
     │                                           ▼
     │                                    ┌──────────────┐
     │                                    │ Poll/webhook  │
     │                                    │ for reply     │
     │                                    └──────┬───────┘
     │                                           │ human answers
     │    --resume <session_id> -p "<answer>"    │
     │◄──────────────────────────────────────────┘
     │
     ▼
  Agent continues with full context
```

**Подтверждённые факты:**
- `AskUserQuestion` denial НЕ зависит от `--dangerously-skip-permissions` (denied в обоих режимах — причина: нет терминала в `-p` режиме)
- Вопрос **структурирован** (question, header, options[], multiSelect) — можно рендерить в GitHub/Slack/email
- `--resume <id> -p "<answer>"` подаёт ответ как новый user message — Claude видит свой предыдущий вопрос и понимает контекст
- Стоимость: ~$0.05 за вопрос (3 API turns) + ~$0.03 за resume = ~$0.08 за один HITL roundtrip

**Оставшиеся вопросы:**
- Как движок извлекает данные: парсить `permission_denials` из финального `result` события (проще) или стримить и ловить `tool_use` (real-time)?
- Нужен ли `--no-session-persistence` для контроля над хранением сессий?
- Как обрабатывать timeout при ожидании ответа от человека (session TTL)?
- Работает ли этот паттерн когда агент вызывает AskUserQuestion в середине сложной задачи (не как первое действие)?

### Сравнение: реализация скрипта рендеринга/доставки вопроса

Задача: взять JSON из `permission_denials`, отрендерить в markdown, опубликовать через `gh issue comment`, позже проверить ответ.

#### Вариант A: Shell-скрипт

```bash
.sdlc/scripts/hitl-ask.sh --issue 42 --run-id abc --node-id pm --question-json '{...}'
.sdlc/scripts/hitl-check.sh --issue 42 --run-id abc --node-id pm
```

- **Плюсы:**
  - Нулевая стоимость (нет API-вызовов Claude)
  - Мгновенный запуск (~50ms)
  - Полностью детерминирован — jq + gh, без вариативности
  - Легко тестировать (`bats`, mock `gh`)
  - Нет зависимости от AGENTS.md, CLAUDE.md, промптов проекта
  - Идеален для шаблонной работы: JSON → markdown → gh api
- **Минусы:**
  - jq-шаблонизация сложных вопросов (multiSelect, nested options) — громоздко
  - Нет "интеллекта" — если формат вопроса изменится, скрипт сломается
  - Парсинг ответа человека — только regex/exact match, нет понимания контекста

#### Вариант B: Агент Claude (запертый в папке, без доступа к проекту)

```bash
claude -p "..." --add-dir /tmp/hitl-sandbox --dangerously-skip-permissions
```

- **Плюсы:**
  - Может "понять" вопрос и красиво отформатировать
  - Может парсить свободную форму ответа человека
  - Гибкий к изменениям формата — адаптируется без правки кода
- **Минусы:**
  - ~$0.02-0.05 за каждый вызов (рендеринг) + за проверку ответа
  - ~5-15 секунд задержка (API roundtrip)
  - Недетерминирован — может отформатировать по-разному каждый раз
  - Всё ещё загрузит CLAUDE.md из `~/.claude/` (user-level settings) — не полностью изолирован
  - Overkill: задача шаблонная, LLM не нужен

#### Вариант C: Агент Claude (полный доступ к проекту)

```bash
claude -p "..." --output-format json
```

- **Плюсы:**
  - Может учитывать контекст проекта (issue body, предыдущие комментарии)
  - Может адаптировать стиль вопроса под проект
- **Минусы:**
  - Читает AGENTS.md, CLAUDE.md, SRS, SDS — тратит токены на нерелевантный контекст
  - ~$0.05-0.10+ за вызов (больше контекста = дороже)
  - Может начать "помогать": отвечать на вопрос вместо пользователя, редактировать файлы, выполнять команды
  - Риск побочных эффектов (особенно с `--dangerously-skip-permissions`)
  - Самый медленный вариант

#### Сводка

| Критерий | Shell-скрипт | Агент (sandbox) | Агент (проект) |
|---|---|---|---|
| Стоимость | $0 | ~$0.03 | ~$0.08 |
| Скорость | ~50ms | ~10s | ~15s |
| Детерминизм | Полный | Низкий | Низкий |
| Гибкость формата | Низкая | Высокая | Высокая |
| Парсинг ответа | Regex | Понимание | Понимание |
| Побочные эффекты | Нет | Минимальные | Возможные |
| Сложность реализации | Средняя (jq) | Низкая (промпт) | Низкая (промпт) |

**Рекомендация:** Shell-скрипт для рендеринга/публикации (шаблонная задача, детерминизм важен). Если позже понадобится "умный" парсинг ответа свободной формы — добавить агента (sandbox) только для этапа check.

### Скрипты пайплайна (`.sdlc/scripts/`)

Скрипты — часть пайплайна, не часть движка. Движок вызывает их через хуки (`after` hook ноды). Скрипты знают про GitHub, движок — нет.

#### `hitl-ask.sh` — опубликовать вопрос в issue

```bash
# Вызов движком (через after-hook при обнаружении permission_denial AskUserQuestion):
.sdlc/scripts/hitl-ask.sh \
  --repo owner/repo \
  --issue 42 \
  --run-id 20260310-abc \
  --node-id pm \
  --question-json '{"questions":[{"question":"...","options":[...]}]}'

# Что делает:
# 1. jq рендерит question-json → markdown (вопрос + нумерованные варианты)
# 2. Добавляет маркер <!-- hitl:20260310-abc:pm -->
# 3. gh issue comment --repo owner/repo -R 42 --body "$markdown"
# 4. exit 0

# Выход (markdown в issue):
# ---
# 🔄 **Агент `pm` ожидает ваш ответ**
#
# > Какой язык программирования использовать?
#
# 1. **Python** — Универсальный язык
# 2. **Go** — Быстрый компилируемый
# 3. **TypeScript** — Типизированный JS
#
# _Ответьте комментарием ниже._
# <!-- hitl:20260310-abc:pm -->
# ---
```

#### `hitl-check.sh` — проверить наличие ответа

```bash
# Вызов движком (при resume, через resume_trigger скрипт):
.sdlc/scripts/hitl-check.sh \
  --repo owner/repo \
  --issue 42 \
  --run-id 20260310-abc \
  --node-id pm \
  --bot-login "my-bot[bot]"

# Что делает:
# 1. gh api "repos/{owner}/{repo}/issues/42/comments" --paginate
# 2. jq: найти комментарий с маркером <!-- hitl:20260310-abc:pm -->
# 3. jq: взять его .id и .created_at
# 4. jq: найти первый комментарий ПОСЛЕ маркера где .user.login != bot-login
# 5. Если найден → вывести .body в stdout, exit 0
# 6. Если нет → exit 1 (ответа ещё нет)
```

#### Интеграция с движком

```yaml
# pipeline.yaml — пример конфигурации
defaults:
  hitl:
    ask_script: .sdlc/scripts/hitl-ask.sh
    check_script: .sdlc/scripts/hitl-check.sh
    poll_interval: 60  # секунд между проверками
    timeout: 7200      # 2 часа макс ожидания
```

Движок при обнаружении `permission_denials` с `AskUserQuestion`:
1. Вызывает `ask_script` с параметрами
2. Сохраняет `session_id` + `node_id` в `state.json` (статус `waiting`)
3. Входит в цикл: `sleep poll_interval` → `check_script` → если exit 0, resume
4. При timeout → узел fail
